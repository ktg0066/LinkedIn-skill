## ğŸ”„ Pull Request Process

### Before Submitting

1. **Fork the repository** and create your branch from `main`
2. **Test your changes** thoroughly
3. **Update documentation** if you've changed functionality
4. **Follow the code style** (see below)
5. **Write clear commit messages**

### Commit Message Format

Use clear, descriptive commit messages:

```
<type>: <subject>

<body>

<footer>
```

**Types:**
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `style`: Formatting, missing semicolons, etc.
- `refactor`: Code restructuring
- `test`: Adding tests
- `chore`: Maintenance

**Examples:**
```
feat: Add carousel template for framework posts

Added new template in /templates/ for creating LinkedIn carousel 
content from frameworks. Includes 10-slide structure and design tips.

Closes #42
```

```
docs: Clarify voice sample requirements

Updated voice-samples README to specify minimum 500 words per sample
and added examples of good vs bad samples.
```

### Pull Request Checklist

- [ ] My code follows the project's style guidelines
- [ ] I have performed a self-review of my changes
- [ ] I have commented my code where necessary
- [ ] I have updated the documentation accordingly
- [ ] My changes don't break existing functionality
- [ ] I have tested my changes in Claude Desktop/Web
- [ ] I have anonymized any personal/company data

### Review Process

1. **Maintainers review** within 7 days
2. **Feedback provided** if changes needed
3. **Approval** when ready
4. **Merge** into main branch
5. **Credit** added to CHANGELOG.md

## ğŸ“ Style Guidelines

### Markdown Files

- Use **clear headings** (##, ###)
- Keep **line length under 100 characters** where practical
- Use **code blocks** with syntax highlighting
- Include **examples** for complex concepts
- Add **emojis sparingly** for visual navigation

### File Naming

- Use `kebab-case` for file names: `my-template.md`
- Use descriptive names: `framework-template.md` not `template1.md`
- Group related files in appropriate directories

### Template Structure

When creating new templates:

```markdown
# [Template Name]

## Overview
[Brief description of when to use this template]

## Structure
[The actual template with placeholders]

## Why This Works
[Explanation of the psychology/strategy]

## Examples
[Real examples of this template in action]

## Update Log
[Track changes over time]
```
